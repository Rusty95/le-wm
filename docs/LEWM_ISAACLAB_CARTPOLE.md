# LeWM IsaacLab Cartpole

This page records our IsaacLab Cartpole deployment built on top of LeWM. It is intentionally separate from the upstream LeWM README, because this is an applied reproduction and control experiment rather than part of the original paper documentation.

## Result Preview

The current best path is:

```text
IsaacLab RGB camera
  -> LeWM encoder
  -> last 3 latent embeddings
  -> lightweight latent policy head
  -> Cartpole action
```

PPO is only used as an offline data generator. It is not loaded during online deployment.

## Quick Start

Run the current best online controller in IsaacLab:

```bash
source /home/hall/miniconda3/etc/profile.d/conda.sh
conda activate env_isaaclab
export PYTHONPATH=/home/hall/code/le-wm:/home/hall/code/RL-Learning-BasedOn-IsaacLab/source/rl_lab_learning:${PYTHONPATH:-}
export LD_PRELOAD=${LD_PRELOAD:+$LD_PRELOAD:}/lib/aarch64-linux-gnu/libgomp.so.1
cd /home/hall/code

python le-wm/scripts/isaaclab_lewm_policy_cartpole.py \
  --task RLLab-Cartpole-SwingUp-RGB-Camera-Direct-v0 \
  --checkpoint /home/hall/code/.stable-wm/checkpoints/lewm_full_angle_multistep_h10/weights_epoch_100.pt \
  --policy-head /home/hall/code/.stable-wm/checkpoints/lewm_full_angle_latent_policy.pt \
  --action-stats-h5 /home/hall/code/.stable-wm/datasets/isaaclab_full_angle_120k.h5 \
  --episodes 1 \
  --episode-len 300 \
  --episode-length-s 25 \
  --initial-pole-angle-range -0.25 0.25 \
  --high-contrast-scene \
  --save-gif \
  --gif-out /home/hall/code/.stable-wm/visualizations/lewm_latent_policy_near_upright.gif \
  --out /home/hall/code/.stable-wm/eval/lewm_latent_policy_near_upright.json \
  --seed 9317 \
  --device cuda:0
```

## Why This Route

We tried several LeWM-only control variants before settling on the current method:

```text
1. LeWM + MPC with hand-written cost
2. LeWM + state probe + CEM
3. LeWM + velocity probe
4. LeWM + rollout-aware velocity probe
5. LeWM encoder + latent policy head
```

MPC and probe-based control could survive short tests, but they often drifted to the track edge, oscillated near boundaries, or failed to produce strong recovery actions after large disturbances. The latent policy head is simpler, faster, and currently gives the most stable online behavior.
